# Agent integration is the CLI run by the harness, not an MCP/async-job surface

For coding-agent integration we deliberately ship **no MCP server and no agent-facing job protocol**. An agent uses getdocs by running the ordinary CLI: synchronously for a bounded fetch ("get me this docs section"), or as a background task that the agent's own harness resumes on completion for a whole-site mirror. The CLI's existing `out/` tree + `crawl.json` Manifest *is* the return value; the only addition agents need is a machine-readable completion summary (a final stderr line plus opt-in `--summary-json`) carrying the run's **Outcome** (`crawled` vs `cloned`, page count, output path, truncation).

We chose this over an MCP server after two findings. First, the MCP value proposition collapses on return shape: an agent can't absorb a multi-page crawl inline (token budget), so the useful return is "a path it can grep" — which a local CLI already gives for free, while a remote MCP server could not. Second, the "fire-and-forget, wake me when the crawl finishes" pattern is **not** an MCP capability: MCP tool calls are request/response, and as of early 2026 only Claude Code auto-resumes an agent on background-command completion (and only while the session stays open). Every other surveyed agent (Codex, Cursor, Copilot, Cline, Windsurf, Gemini CLI, Aider, Devin) is poll-or-block. So the background-notify UX we want lives in the harness's CLI/background layer, not in MCP.

Consequences: the **synchronous** CLI is the portable baseline that works under every agent; **background-with-resume** is a Claude-Code-only bonus; agents on other harnesses fall back to blocking or polling getdocs's known output path. Distribution to agents is therefore a Claude Code **plugin** (a `/getdocs` command that bootstraps via `uvx`, picks sync vs background, and reads the Outcome summary to guide next steps) rather than a tool-registry listing. If a hosted/remote agent use case ever becomes primary, this decision should be revisited — that's the case an MCP server with server-side retrieval would actually serve.
